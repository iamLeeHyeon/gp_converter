# 기타 탭 PDF 인식 설계

작성일: 2026-06-24

## 목표

기타 탭(TAB) 표기가 포함된 디지털 PDF를 업로드하면, 탭에 명시된 정확한 현/프렛 정보를 반영한 `.gp5`로 변환한다. 기존 MVP(표준악보 PDF → `.gp5`)는 그대로 유지하고, 탭 PDF가 들어왔을 때만 추가 경로를 태운다.

## 범위

- **입력:** 디지털 탭 PDF — 표준악보(리듬 표기용 5선보)와 탭보표(6선보)가 같은 시스템에 같이 있는 형태(Guitar Pro/MuseScore 등에서 출력한 일반적인 듀얼 표기). 실제 샘플(`Santa Tell Me.pdf`, Ariana Grande)로 구조 확인함.
- **범위 밖(이후 확장):** 스캔된 탭 이미지, 탭보표만 있고 표준악보가 없는 PDF, 표준 외 운지(코드 다이어그램, 해머링/슬라이드 등 기법 기호), 다중 보이스/동시 화음 탭의 정밀 정렬.

비목표(YAGNI): 알 수 없는 튜닝 자동감지(현 개수=6 가정), 다중 트랙 탭 동시 처리.

## 접근법

**핵심 아이디어:** 기존 MVP 파이프라인(Audiveris → music21 → PyGuitarPro)은 손대지 않는다. Audiveris는 표준악보(리듬 표기) 줄에서 음정·박자를 정확히 뽑아내는 걸 이미 확인했다(이전 디버깅에서 동일 PDF로 320음표 정확히 추출됨). 탭보표는 Audiveris가 다루지 못하는 부분이므로, **탭보표에서 (현, 프렛)만 별도로 읽어 기존 음정·박자 순서열에 순서대로 매칭**해 끼워넣는다.

```
PDF ──┬─→ Audiveris(기존) → MusicXML → music21 ──→ (음정, 박자) 순서열
      │                                                     │
      └─→ [신규] tab_reader(pdfminer.six)            ──→ (현, 프렛) 순서열
                                                              │
                                     순서대로 1:1 매칭 ────────┘
                                              │
                          PyGuitarPro(.gp5 쓰기) — 매칭 성공 시 명시적 (현,프렛) 사용,
                          실패/불일치 시 기존 휴리스틱(최저프렛) 폴백
```

## 탭보표 자동 감지

PDF의 수평 벡터선(`LTLine`/`LTRect`, `pdfminer.six`)을 y좌표로 등간격 클러스터링한다. **간격의 절대값은 소프트웨어마다 다르므로 쓰지 않고, 연속된 줄의 개수만 본다**:
- 연속 6줄 → 탭보표
- 연속 5줄 → 표준악보(참고용, 직접 파싱하지 않음)

실제 샘플(줄 간격 ~10.2pt)과 LilyPond로 생성한 합성 픽스처(줄 간격 ~7.5pt)에서 절대 간격은 다르지만 "탭=6줄/표준악보=5줄" 패턴은 동일하게 확인됨 → 개수 기반 판단이 소프트웨어 간 차이에 안전함이 검증됨.

탭보표가 하나도 검출되지 않으면 기존 MVP 흐름 그대로 진행한다(이 설계는 추가 경로일 뿐, 기존 동작을 변경하지 않음).

## 컴포넌트

**신규 모듈: `app/pipeline/tab_reader.py`**

1. `detect_tab_staves(pdf_path) -> list[TabStaffRegion]`
   페이지별로 6줄 그룹을 찾아 영역(페이지, y범위, 줄 좌표)을 반환한다. 없으면 빈 리스트.

2. `extract_tab_notes(pdf_path, regions) -> list[TabNote]`
   각 영역의 y범위에 속한 글자(`LTChar`, pdfminer 기본 텍스트 그룹화로 숫자 뭉치 단위 결합)를 추출한다.
   - 숫자의 y좌표 → 영역 내 6줄 중 가장 가까운 줄에 매칭 → 현 번호(위=1번현~아래=6번현, 표준 탭 표기 관례)
   - `X`(뮤트 표시)는 스킵한다 — v1 한계로 문서화, 탭 순서열에도 포함하지 않음
   - 보표 내에서 좌→우, 문서 내에서 위→아래 순서로 정렬해 시간순 리스트로 반환

3. **정렬 함수** (`app/pipeline/musicxml_to_gp.py` 또는 신규 작은 모듈에 추가):
   music21에서 뽑은 음표 개수와 탭에서 뽑은 (현,프렛) 개수가 **정확히 일치할 때만** 1:1 순서대로 매칭해 PyGuitarPro 작성 시 명시적 현/프렛을 사용한다. 개수가 다르면 매칭을 시도하지 않고 탭모드 전체를 포기, 기존 휴리스틱(최저프렛 배치)으로 완전 폴백한다(부분 매칭은 하지 않음 — 틀리게 매칭되는 위험보다 안전한 폴백이 낫다는 판단).

**오케스트레이터(`orchestrator.py`) 변경:** `run_conversion`이 `detect_tab_staves`로 탭 여부를 확인하고, 있으면 `extract_tab_notes` + 정렬을 거쳐 `musicxml_to_gp5`에 명시적 현/프렛 힌트를 넘긴다. 없거나 정렬 실패 시 기존 인자만으로 호출(현재 동작과 동일).

## 장점

탭에서 현/프렛을 직접 읽으므로 **튜닝이 달라도 정확하다**(드롭D 등 — 단, v1은 6줄 여부만 보고 탭보표를 식별하며 줄 개수=현 개수 6 가정은 유지). 기존 휴리스틱은 표준튜닝을 가정한 최저프렛 추정이라 이 경로가 한 단계 더 정확하다.

## 에러 처리

전부 "탭모드 실패 → 기존 동작으로 폴백" 원칙. 절대 변환 자체를 실패시키지 않는다.

- 탭보표 미검출 → 기존 흐름(현재 MVP와 동일)
- 탭 검출됐으나 숫자 추출 실패/개수 불일치 → 탭모드 포기, 휴리스틱 폴백
- `pdfminer.six` 파싱 예외 → 탭모드 포기, 휴리스틱 폴백(Audiveris 단계는 영향 없음)

## 테스트 (TDD)

- **단위:** `detect_tab_staves`, `extract_tab_notes`를 LilyPond로 만든 합성 듀얼(표준악보+탭) PDF 픽스처로 검증(저작권 무관, 생성 가능 확인됨: `\new Staff`와 `\new TabStaff`를 함께 둔 `.ly` → PDF).
- **정렬 단위테스트:** 가짜 음정 시퀀스 + 가짜 탭 시퀀스로 매칭 로직만 검증(개수 일치/불일치 양쪽 케이스).
- **통합:** 합성 PDF로 전체 파이프라인 1회 실행(`pytest -m integration`, 실제 Audiveris 사용).
- 실제 샘플(`Santa Tell Me.pdf`, 저작권물)은 **커밋하지 않고** 로컬 수동 검증에만 사용한다.

## 한계 (v1, 문서화)

- `X`(뮤트/타격) 표시는 스킵 — GP5에 음표로 들어가지 않음
- 화음(동시발음 탭 숫자 여러 개)의 정밀 정렬은 v1 범위 밖 — 개수 불일치로 처리되어 전체 폴백될 가능성 높음
- 해머링/슬라이드/벤딩 등 기법 기호는 무시(프렛 숫자만 읽음)
- 탭보표만 있고 표준악보(리듬용 5선보)가 없는 PDF는 박자 정보를 얻을 수 없어 v1에서 미지원(범위 밖)
- 스캔 이미지 탭은 미지원(범위 밖, 이후 PaddleOCR 등 검토 대상)
