# 다중 파트 → 다중 트랙 변환 설계

## 목표

지금까지 표준악보 변환 경로(`app/pipeline/musicxml_to_gp.py`)는 MusicXML의 첫 번째
`<part>`만 읽고 나머지는 버렸다(`score.parts[0]` 고정). 기타 듀엣/앙상블처럼 여러
파트로 된 악보를 변환하면, GP5 결과물이 항상 파트 1개짜리(첫 파트만)로 나온다.

이 작업은 MusicXML의 모든 파트를 각각 GP5 트랙으로 변환한다. 대상은 **모든 파트가
기타인 악보**(듀엣/앙상블)로 한정한다 — 보컬/피아노 등 다른 악기가 섞인 리드시트는
스코프 밖이다(모든 트랙이 표준 6현 어쿠스틱 기타 튜닝을 그대로 쓴다).

## 스코프 경계 (명시적으로 하지 않는 것)

- **탭인식(OMR)은 확장하지 않는다.** 지금처럼 탭보표가 감지되면 첫 파트 기준
  단일트랙 tab-OMR 경로(`token_texts_to_gp5`)를 그대로 탄다 — 이 경로는 이번 작업과
  무관하게 완전히 그대로 둔다. 다중트랙은 **탭이 감지되지 않은 순수 표준악보
  경로에만** 적용된다.
- **파트별 악기 인식은 하지 않는다.** 모든 트랙이 표준 어쿠스틱 기타(6현 EADGBE,
  기본 음색)로 고정된다. 파트 이름/악기 메타데이터로 트랙마다 다른 튜닝·음색을
  주는 건 스코프 밖(지난 세션에 "오로지 어쿠스틱 기타만" 결정과 일관).
- 파트가 1개뿐인 기존 악보는 지금과 동일한 결과가 나와야 한다(회귀 없음).

## 아키텍처

### 곡 전체 공통 vs 트랙별 분리

MusicXML의 정보는 두 층위로 나뉜다:

- **곡 전체 공통** (GP5 `Song.measureHeaders`, 모든 트랙이 공유): 박자표, 조표,
  반복표 열기/닫기, volta(엔딩), D.C./D.S./Coda/Segno 지시, 최초 템포와 곡중간
  템포 변화, 곡 제목/아티스트. 이 전부를 **파트 0 기준**으로 한 번만 계산한다 —
  같은 곡의 여러 파트가 서로 다른 마디 구조(박자표 등)를 갖는 일은 실무상 없다고
  가정한다.
- **트랙별 독립**: 음표/화음/쉼표, 아티큘레이션, 벤드/트릴/비브라토/슬라이드/
  하모닉/고스트/운지법, 코드심볼, 가사. 파트마다 따로 추출해서 각자의 GP5
  `Track`에 담는다.

### 코드 변경 지점

1. **`_collect_notes`를 파트 파라미터화**: 현재 `_collect_notes(score, xml_path)`가
   내부에서 `part = score.parts[0]`로 고정하는 걸, `_collect_notes(part, xml_path,
   part_index)`처럼 명시적으로 파트를 받도록 바꾼다. **모드 분기는 만들지 않는다**
   — 모든 파트에서 지금과 똑같이 `MeasureData`(박자표/조표/반복표/direction 포함)를
   그대로 추출한다. "파트 0만 곡 전체에 반영된다"는 규칙은 이 함수 안이 아니라
   `_build_song`이 마디 헤더를 조립하는 단계에서 지킨다(아래 4번) — 즉
   `measures_data_by_track[0]`의 필드만 `song.measureHeaders`에 쓰고,
   `measures_data_by_track[1:]`의 동일 필드는 무시한다. 이렇게 하면 `_collect_notes`
   자체는 지금처럼 파트 하나만 알면 되는 단순한 함수로 남는다.

2. **`_scan_raw_technicals`에 파트 인덱스 추가**: 지금은 첫 `<part>` 엘리먼트만
   본다(최근 커밋 `9105299`). 여러 파트를 지원하려면 `part_index`를 받아
   `root.findall("part")[part_index]`로 스캔 대상을 바꾼다. MusicXML은 `<part-list>`
   순서와 `<part>` 문서 순서가 같다는 전제(스펙상 요구되고 Audiveris도 지킴) —
   `score.parts[i]`와 `root.findall("part")[i]`가 같은 파트를 가리킨다고 가정한다.

3. **`_build_hairpin_velocities`/`_build_tempo_changes`**: 이미 `part`를 인자로
   받으므로 시그니처 변경 없음. 다만 템포 변화는 "곡 전체" 개념이라 **파트 0
   호출 결과만 사용**하고, 다른 파트에서 호출한 결과는 버린다(트랙마다 따로
   재생 속도가 달라질 수 없으므로).

4. **`_build_song`을 파트 리스트로 확장**: 현재 `_build_song(measures_data:
   List[MeasureData], tab_hints)` → `_build_song(measures_data_by_track:
   List[List[MeasureData]], tab_hints)`로 시그니처 변경. `gpm.Song()`은 트랙
   1개로 시작하므로, 파트 개수만큼 `song.tracks.append(gpm.Track(song))`로
   추가하고 각 트랙에 표준 튜닝을 설정한 뒤, 트랙별로 기존 마디/보이스/음표
   빌드 로직을 반복한다. `tab_hints`는 첫 트랙에만 적용 가능(기존 동작 유지,
   다른 트랙은 항상 휴리스틱).

5. **`musicxml_to_gp5` 오케스트레이션 변경**: `score.parts`를 순회하며 파트마다
   `_collect_notes`(+`_scan_raw_technicals`)를 호출해 `measures_data_by_track`을
   조립하고, 파트 0에서 얻은 곡 전체 공통 정보(템포/메타데이터/마디헤더 관련)를
   그대로 적용한다.

### 에러 처리

파트 하나를 처리하다 예외가 나면(예: 그 파트만 이상한 구조) 전체 변환을 실패시킬지,
그 파트만 건너뛸지 결정 필요 — **전체 실패로 처리한다**(부분적으로 깨진 다중트랙
결과보다 명확한 에러가 낫다는 기존 프로젝트 관례와 일관, `GpConvertError` 그대로
전파).

### 테스트 계획

- 파트 1개짜리 기존 fixture들 — 전부 회귀 없이 그대로 통과해야 함(가장 중요한
  안전망).
- 파트 2개짜리(둘 다 표준악보, 서로 다른 음표) MusicXML fixture 새로 작성 —
  `song.tracks`가 2개이고 각각 다른 음표를 담고 있는지 검증.
- 파트 2개 중 하나에만 벤드/트릴 등 이펙트가 있는 경우 — 그 이펙트가 해당
  트랙에만 적용되고 다른 트랙에 안 새는지 검증(정확히 최근 고친 "파트 오염"
  버그의 반대 상황을 다중트랙 버전으로 재확인하는 셈).
- 파트 2개 중 하나에 탭보표가 있는 경우 — 기존 tab-OMR 단일트랙 경로가 그대로
  타는지(다중트랙 경로로 새지 않는지) 확인.
- 템포/마디헤더(박자표 등)가 파트 0 기준으로 모든 트랙에 동일하게 반영되는지
  확인.
